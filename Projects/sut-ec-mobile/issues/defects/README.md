# 不具合(defects)起票フォルダ

テスト実施(自動シナリオ含む)で検出した不具合の起票先。**1件=1ファイル**で管理する
([テスト計画書 §7](../../docs/testbases/test/test-plan.md) の成果物「不具合一覧」)。

## 起票ルール(標準)

- **1不具合につき1ファイル**を本フォルダに作成する(詳細=個別ファイル)。
- ファイル名: `D-<連番2桁>-<英小文字ケバブの短い要約>.md`(例 `D-01-cart-qty1-no-auto-delete.md`)。
  連番は既存ファイルの最大+1。
- **状態の一覧は [INDEX.md](INDEX.md)**(対応状況ダッシュボード)。起票・状態変更したら**個別ファイルと
  INDEX.md の両方を更新**する(集計も合わせる)。
- **仕様(testbase の SC/TC)に従っていない挙動は、テストを実挙動に寄せて緑にせず、仕様どおりの
  期待のまま RED(失敗)にして追跡する**。緑で隠さない。対応するシナリオ名を本ファイルに記す。
- 修正されたら状態を「修正済み」にし、対応テストが緑になることを確認する。却下(仕様確定)なら
  理由を記し、テスト側の期待を確定した仕様へ更新する。
- テスト文書レビューで既出の指摘([test-review-record.md](../../docs/testbases/test/test-review-record.md) の C/V/R/F)は
  そちらで管理し、ここには**テスト実施で新規検出したもの**を起票する(必要なら相互参照)。

## テンプレート

新規起票時は下記をコピーして `D-<n>-<slug>.md` を作る。

```markdown
# D-<n> <一文タイトル>

| 項目 | 内容 |
|---|---|
| 対象 | <画面/機能> |
| 対応 SC/TC | SC-xx / TC-xx |
| 重大度 | 高 / 中 / 低 |
| 状態 | 起票 / 修正中 / 修正済み / 却下(仕様確定) |
| 検出 | <自動シナリオ名 or 手動> / <端末・OS> |
| 起票日 | YYYY-MM-DD |

**再現手順**
1. …

**期待結果(SC/TC 根拠)**: …

**実際の挙動**: …

**証拠**: <ft_snapshot 抜粋 / レポートパス / スクショ>

**仕様裁定(要判断があれば)**: …
```

## 重大度・状態の凡例

- 重大度: 高(業務停止・データ不整合)/ 中(機能不整合・代替手段あり)/ 低(表示・軽微)
- 状態: 起票 / 修正中 / 修正済み / 却下(仕様確定)
